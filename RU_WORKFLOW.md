# WORKFLOW.md

\> Описание стандартного процесса разработки в команде проекта.

\> Документ отражает рутину и процессы, необходимые для поддержания высокого качества кода, истории изменений, воспроизводимости результатов и прозрачности командной работы.

## 🗂️ 1. Постановка задачи

### Кто заводит задачи?
- **Ментор / Технический лидер / Участник, нашедший проблему**.

### Где?
- Используется **таск-доска** (например, GitHub Projects, Linear, Trello, или Issue-трекер в GitLab).
- Категории задач:
  - `Feature`: новая функциональность;
  - `Fix`: исправление багов;
  - `Refactor`: архитектурные изменения без изменения логики;
  - `Docs`: улучшение документации;
  - `Tests`: покрытие тестами.

### Как оформляется задача?
Каждая задача (Issue) содержит:
- **Краткое описание**;
- **Ожидаемое поведение**;
- **Критерии приёмки (Acceptance Criteria)**;
- **Тег приоритета** (`high`, `medium`, `low`);
- **Связь с другими задачами** (если применимо);
- **Ответственный за постановку**.

## 🧑‍💻 2. Взятие задачи в работу

### Кто берёт?
- Любой участник команды, по договорённости или назначению.

### Процесс:
1. Меняет статус задачи на `In Progress`;
2. Создаёт **feature-ветку** в git:
```
git checkout -b feature/<short-keyword>-<brief-description>
```

> Пример: `feature/parser-cli`, `fix/header-overflow`, `refactor/db-schema`.

## 🛠️ 3. Разработка

### Что входит:
- Исходный код;
- Юнит-тесты (обязательно!);
- Документация кода через **Doxygen-style комментарии** (если C/C++/Python);
- Обновление/создание `README.md` и `USAGE.md` при необходимости;
- Промежуточные коммиты с осмысленными сообщениями.

### Документирование кода:
- Используем формат **Doxygen**:
```c
/**
* @brief Парсит строку и возвращает результат
* @param input строка
* @return результат парсинга
*/
Result parse(const std::string& input);
```

## 🧪 4. Тестирование

### Кто пишет тесты?

* **Разработчик, реализующий модуль**.
* Возможно участие менторов или ревьюеров при сложных кейсах.

### Где хранятся?

* `/tests/module_name/...`
* Названия тестов отражают поведение и граничные случаи.

## 📝 5. CHANGELOG

### Кто обновляет?

* **Автор фичи / фиксa**.

### Когда?

* Перед созданием Pull Request-а.

### Как?

* Вручную или с помощью CLI-утилиты [`git-cliff`](https://github.com/orhun/git-cliff) (или аналога).

#### Формат:

```markdown
## [1.2.0] - 2025-05-31
### Added
- Поддержка CLI-режима для парсера

### Fixed
- Устранена утечка памяти в модуле анализа

### Changed
- Улучшено логирование входящих данных
```

## 🚀 6. Pull Request (PR)

### Кто создаёт?

* Автор задачи.

### Что должен содержать PR:

* Связь с задачей (`Closes #123`);
* Список основных изменений;
* Скриншоты (если есть визуальные изменения);
* Уверенность, что код покрыт тестами;
* Уверенность, что `CHANGELOG.md` обновлён.

### Стандарты:

* Проходит CI (если настроен);
* Минимум 1 одобрение ревьюера.

## 👀 7. Code Review

### Кто ревьюит?

**Назначенный ревьюер** или любой другой участник (по очереди или специализации).

### Цель:

* Проверить читаемость, архитектуру, потенциальные баги;
* Проверить тесты;
* Проверить соответствие `STYLE.md`, если он есть.

### После ревью:

* Автор либо исправляет замечания, либо отвечает по ним;
* После одобрения — мерж в `main`/`develop`.

## 📦 8. Мерж и релиз

### Кто мержит?

* Автор (если самодостаточная задача) или ревьюер (если сложная фича).

### Что происходит после мержа:

* Запускаются CI-сборки и тесты (если настроены);
* Версия инкрементируется вручную или автоматически (`patch`, `minor`, `major`);
* Если нужен публичный релиз, тегируется и создаётся `Release` с описанием.

## 🔁 9. Архивация / демонстрация проектов

### Проблема:

* Основной репозиторий приватный, и через 1.5 года сервер может быть очищен.

### Решение:

* Заводим **внешний публичный репозиторий**, содержащий:

  * итоговый код;
  * документацию;
  * README без прямых формулировок задания;
  * CI (если есть);
  * CHANGELOG;
  * ссылку на внешний профиль (например, профиль студента на платформе 21 School).

## 🧩 Дополнительные правила

* **Формат коммитов**: `type(scope): message`, например `fix(parser): handle empty input`.
* **Субмодули**: допустимы, если нужно показать код снаружи.
* **Чувствительные данные**: никогда не коммитим `*.key`, `*.pem`, IP, логины, или структуру заданий.

## 🧠 Используемые инструменты

| Название                             | Цель                          |
| ------------------------------------ | ----------------------------- |
| `git`                                | Контроль версий               |
| `make`                               | Сборка проекта                |
| `ctest` / `pytest`                   | Тестирование                  |
| `doxygen`                            | Документирование кода         |
| `git-cliff`                          | Генерация CHANGELOG           |
| `CI/CD` (GitHub Actions / GitLab CI) | Автоматизация сборки и тестов |
